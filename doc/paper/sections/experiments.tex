\section{Experiments}
\label{sec:experiments}

This section presents an experimental evaluation of the toolchain on the Franka FR3 robot arm.

% ---- A. Research Questions ----
\subsection{Research Questions}

We define five research questions that guide the experimental design:

\begin{itemize}
    \item \textbf{RQ1 (Coverage):} Does the generated geometry preserve coverage of all robot link meshes across all modes and presets?
    \item \textbf{RQ2 (Mode comparison):} How do convex, sphere, and capsule approximation modes trade off primitive count and geometric tightness on a representative robot model?
    \item \textbf{RQ3 (Capsule ablation):} How sensitive is capsule quality to section count, capsule budget, adaptive circle fitting, and splitting thresholds?
    \item \textbf{RQ4 (Runtime):} What computational cost does the full URDF approximation pipeline impose for each mode and preset?
    \item \textbf{RQ5 (Mesh source):} How do visual-mesh and collision-mesh sources affect the fitting quality and metrics?
\end{itemize}

% ---- B. Dataset ----
\subsection{Dataset}

The primary test platform is the Franka FR3 robot arm \cite{franka2020fr3}, consisting of 8 arm links (link0--link7) plus hand and finger links.
The FR3 robot model is bundled with the repository and includes both visual meshes (COLLADA format files representing the true link geometry) and collision meshes (pre-simplified STL files).
We evaluate on the 8 arm links, which contain the richest geometry variation.
Robots with greater mesh complexity or more diverse link shapes (e.g., quadrupeds, mobile manipulators) may exhibit different trade-offs and are left for future work.

% ---- D. Experimental Setup ----
\subsection{Experimental Setup}

All experiments are run on a single machine with an Intel Core i7-13700K CPU, 32~GB RAM, and an NVIDIA RTX 4090 GPU (used only for PyBullet visualization, not for computation).
The pipeline is invoked via Docker, using the Python API with visual mesh source unless otherwise noted.
Runtimes are reported as wall-clock time averaged over three runs and include Docker container startup overhead (approximately 1--2~s per invocation).
The capsule union volume is estimated with $S = 64$ samples per axis (twice the default of $S = 32$) during validation to improve estimate accuracy, while generation uses the default for speed.

\textbf{Convex mode} produces one convex hull per link (8 primitives total) and is the fastest configuration at 0.5~s.
Because the convex hull encloses the original mesh vertices by construction, coverage is guaranteed.
However, the output is mesh-based and does not admit the closed-form distance evaluation available for analytic primitives.

\textbf{Sphere-tree modes} trade primitive count against coverage tightness.
The \emph{single} preset produces one sphere per link (8 total) with near-perfect coverage: only two links exhibit tens-of-micrometers uncovered vertices ($2\times 10^{-5}$~m worst-case).
The \emph{default} preset generates 63 spheres across the 8 links but leaves all links with small uncovered regions (worst-case $1.8\times 10^{-2}$~m).
This behavior is by design --- the sphere tree is a medial-axis approximation that places spheres to match the shape's interior, not to enclose all surface vertices.

\textbf{Capsule modes} output 8--10 primitives with tightness quantified by the union-volume ratio $\mathrm{capV/aabb}$ (worst-case range 1.32--1.47 across presets) and the radius-inflation metric $r$/binMed (worst-case range 1.27--1.50).
The \emph{high\_detail} preset achieves the best worst-case uncovered distance (2.3~mm) and a slightly lower $\mathrm{capV/aabb}$ (1.32 vs.\ 1.47 for default), while keeping the total primitive count at 10 with a slightly different per-link distribution.
Runtime for capsule modes ranges from 12.7~s (single) to 33.0~s (default), with the \emph{high\_detail} preset completing in 21.8~s.

A key finding is that \textbf{no analytic-primitive mode achieves complete (all-vertex) coverage on all arm links}.
Only the convex mesh output guarantees full enclosure.
Analytic primitives inevitably involve trade-offs between tightness and coverage, and the optimal configuration depends on the downstream application's tolerance for small uncovered regions.

Each row varies one parameter from the default configuration ($N=4$, $K_{\max}=12$, adaptive circles disabled, $\rho_{\max}=1.45$, visual mesh source) while holding others fixed.

\textbf{Section count $N$ is the dominant control parameter.}
Increasing the section count from 2 to 8 reduces the worst-case uncovered distance from 6.69~mm to 1.20~mm (5.6$\times$ improvement) at the cost of raising the capsule count from 8 to 11 and increasing $\mathrm{capV/aabb}$ from 1.37 to 1.56.
However, the relationship is not monotonic: the default configuration ($N=4$) exhibits a \emph{worse} worst-case uncovered distance (10.23~mm) than $N=2$ (6.69~mm), because the 4-section fit splits one link into two capsules, creating a coverage gap at the split boundary.
At $N=8$, this gap nearly disappears (to single-digit micrometers).

\textbf{Per-link capsule budget $K_{\max}$ plateaus.}
Values above 4 all produce identical output (10 capsules total, same metrics), indicating that the split logic's improvement threshold ($\delta_{\min} = 5 \times 10^{-3}$) rejects additional capsules as not beneficial regardless of budget.
The budget is therefore not binding at the default value of 12 on this robot.

\textbf{Adaptive circle fitting} enables per-section multi-circle fitting, which dramatically increases the capsule count from 10 to 39 (3.9$\times$) and runtime from 33~s to 57~s.
The worst-case uncovered distance improves from 10.23~mm to 1.35~mm (7.6$\times$), but the radius-inflation metric $r$/binMed spikes to 2.05 on one link, indicating significant axial radius variation within individual capsules.

\textbf{Radius-inflation control $\rho_{\max}$} set to --1 (disabling all splitting) produces metrics identical to the default configuration in which the ratio is active at 1.45.
This confirms that the other pipeline mechanisms (COA threshold, $\delta_{\min}$) already prevent excessive radius variation before $\rho_{\max}$ becomes the active constraint, at least at this threshold value.

\textbf{Mesh source} (visual vs.\ collision) produces identical capsule fits on the FR3, suggesting that the collision meshes (simplified convex envelopes) capture the same geometric structure as the visual meshes for this robot.
This finding may not generalize to robots with greater divergence between visual and collision mesh representations.

\textbf{No configuration achieves full coverage on all arm links.}
Every variant leaves at least one link with a positive worst-case signed distance, confirming that the coverage-tightness trade-off is fundamental to the capsule representation.

% ---- G. Summary ----
\subsection{Summary of Findings}

The experiments yield the following key findings:
\begin{enumerate}
    \item Convex mode is fastest (0.5~s) and guarantees coverage, but produces only mesh-based output.
    \item Sphere-tree modes provide 8--63 primitives; the single-sphere preset offers near-perfect coverage at minimal primitive count but the default tree does not guarantee enclosure.
    \item Capsule modes deliver 8--11 primitives with quantifiable tightness ($\mathrm{capV/aabb}$ 1.32--1.56, $r$/binMed 1.27--1.50).
    \item The section count $N$ is the dominant capsule-tuning parameter, improving worst-case coverage from 6.7~mm to 1.2~mm as it increases from 2 to 8.
    \item Adaptive circle fitting dramatically increases capsule count (3.9$\times$) but introduces radius inflation.
    \item No analytic-primitive mode achieves full coverage on all links --- this is a fundamental limitation of the representation.
\end{enumerate}

All experimental data, configuration files, and generated figures are stored in the artifact directories corresponding to each experiment request.
